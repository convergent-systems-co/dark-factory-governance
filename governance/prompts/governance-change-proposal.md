# Governance Change Proposal Workflow

This workflow analyzes retrospective aggregation data and persona effectiveness data to propose governance configuration changes. All proposals require human approval — the system never auto-applies governance changes.

## When to Run

Run this workflow periodically (e.g., weekly or after every N governance runs) or when:
- A retrospective aggregation report is available (`governance/schemas/retrospective-aggregation.schema.json`)
- A persona effectiveness report is available (`governance/schemas/persona-effectiveness.schema.json`)
- Multiple review cycles consistently fail or require human overrides

## Inputs

| Input | Schema | Source |
|-------|--------|--------|
| Retrospective aggregation report | `retrospective-aggregation.schema.json` | Generated by retrospective workflow |
| Persona effectiveness report | `persona-effectiveness.schema.json` | Generated from retrospective data |
| Current policy profiles | `governance/policy/*.yaml` | Repository files |
| Current panel defaults | `governance/schemas/panels.defaults.json` | Repository files |
| Threshold tuning policy | `governance/policy/threshold-tuning.yaml` | Repository files |

## Workflow Steps

### Step 1: Load Data

1. Read the most recent retrospective aggregation report
2. Read the most recent persona effectiveness report
3. Read all current policy profiles (`default.yaml`, `fin_pii_high.yaml`, `infrastructure_critical.yaml`, `fast-track.yaml`, `reduced_touchpoint.yaml`)
4. Read panel defaults (`panels.defaults.json`)
5. Read threshold tuning policy (`threshold-tuning.yaml`)

If any required input is missing, skip the analysis that depends on it and note the gap in the output.

### Step 2: Analyze Persona Effectiveness

For each persona in the effectiveness report:

1. **Low signal-to-noise personas** (S/N below `thresholds.low_signal`):
   - Identify the specific finding patterns that are being dismissed
   - Check if the persona definition is too broad or the evaluation criteria are misaligned
   - Propose: weight reduction, persona definition refinement, or removal from specific panels

2. **High signal-to-noise personas** (S/N above `thresholds.high_signal`):
   - Verify the persona has sufficient sample size (`thresholds.minimum_findings`)
   - Propose: weight increase to amplify high-value findings in aggregate confidence

3. **Declining trend personas** (trend = `declining`):
   - Flag for immediate review regardless of current S/N ratio
   - Check for correlation with codebase changes that may have shifted relevance

### Step 3: Analyze Panel Accuracy

From the retrospective aggregation report:

1. **High false positive panels** (`false_positive_rate > 0.30`):
   - Identify which personas contribute most to false positives
   - Propose: confidence threshold increase (require higher confidence to block)
   - Cross-reference with persona effectiveness data

2. **High false negative panels** (`false_negative_rate > 0.10`):
   - These are dangerous — the panel approved changes that later caused incidents
   - Propose: confidence threshold decrease (block more aggressively)
   - Consider adding new personas to the panel

3. **Override frequency analysis**:
   - If a policy profile has override rate > 0.20, the rules may be too restrictive
   - Identify which specific rules are being overridden
   - Propose: rule adjustment or threshold relaxation

### Step 4: Analyze Confidence Thresholds

From the threshold tuning policy and retrospective data:

1. Compare current thresholds to actual panel performance
2. Identify thresholds that are:
   - **Too conservative** — blocking changes that consistently pass post-merge verification
   - **Too permissive** — approving changes that are later reverted or cause incidents
3. Propose specific threshold adjustments with supporting evidence

### Step 5: Generate Proposal

For each identified change, create a structured proposal entry:

```markdown
### Proposal: [Short Description]

**Category:** [persona_weight | panel_threshold | policy_rule | persona_definition]
**Priority:** [high | medium | low]
**Confidence:** [high | medium | low] (based on data quality and sample size)

#### Evidence

- **Data source:** [retrospective report ID / effectiveness report ID]
- **Sample size:** [N runs / N findings]
- **Key metric:** [specific metric value]

#### Current Configuration

```yaml
# File: [path]
[current configuration snippet]
```

#### Proposed Change

```yaml
# File: [path]
[proposed configuration snippet]
```

#### Risk Assessment

- **Impact:** [what changes in governance behavior]
- **Reversibility:** [how to revert if the change causes problems]
- **Blast radius:** [which repos/profiles are affected]
```

### Step 6: Create Proposal Issue

Create a GitHub issue with the collected proposals:

```bash
gh issue create \
  --title "governance-change: [summary of proposals]" \
  --label "governance-change" \
  --body "[structured proposal content]"
```

The issue body should include:
1. **Summary** — Number of proposals, categories, and overall confidence
2. **Data Sources** — Links/references to the reports used
3. **Proposals** — Each proposal in the format from Step 5
4. **Implementation Plan** — If approved, which files change and in what order
5. **Rollback Plan** — How to revert each change

### Step 7: Create Draft PR (Optional)

If the proposals involve specific, well-defined configuration changes:

1. Create a branch: `NETWORK_ID/chore/governance-change-{date}`
2. Apply the proposed changes to the configuration files
3. Create a **draft** PR referencing the proposal issue
4. The draft PR is NOT ready for merge — it exists to show the exact diff

```bash
gh pr create --draft \
  --title "chore: governance config changes from [date] analysis" \
  --body "Draft implementation of proposals from #[issue-number]. DO NOT merge until human review is complete."
```

## Safety Guardrails

1. **Never auto-apply changes** — All proposals require human approval via the GitHub issue
2. **Never modify policy profiles directly** — Changes are proposed as diffs, not applied
3. **Minimum sample size** — Do not propose changes based on fewer than 5 data points
4. **Conservative by default** — When evidence is ambiguous, recommend "no change"
5. **One direction at a time** — Do not simultaneously loosen and tighten thresholds for the same panel
6. **Preserve backward compatibility** — Proposed changes must not break consuming repo configurations
7. **Version bump required** — If a schema or policy profile changes, bump the version field

## Output Format

The workflow produces:
1. A GitHub issue labeled `governance-change` with structured proposals
2. (Optional) A draft PR with the proposed changes implemented
3. A summary comment on the retrospective aggregation issue (if one exists) linking to the proposal

## Integration Points

| Component | Relationship |
|-----------|-------------|
| `retrospective-aggregation.schema.json` | Input — provides panel accuracy and override data |
| `persona-effectiveness.schema.json` | Input — provides per-persona S/N ratios and recommendations |
| `threshold-tuning.yaml` | Input/Output — current thresholds as input, proposed thresholds as output |
| `panels.defaults.json` | Input/Output — current panel config as input, proposed changes as output |
| `governance/policy/*.yaml` | Input/Output — policy rules as input, proposed adjustments as output |
| `startup.md` | Caller — this workflow can be invoked during the startup loop's retrospective step |
